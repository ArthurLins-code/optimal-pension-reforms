0 TL;DR

The pipeline that estimates the welfare value of Brazil's 2015 pension reform lived in one flat folder (trans_retirement/code/), wired together with machine-specific absolute paths and setwd() calls, with no single command to run it. This branch reorganised it — using a team of AI sub-agents under an orchestrator — into a clean build → analysis → presentation layout with a single configuration layer for every path, three master scripts, and a guarded legacy quarantine — and then proved the reorganisation changed zero results by re-running on the 5% sample and hashing every figure and number against a snapshot taken before anything moved.

The keystone idea. Because the "before" and "after" runs use the same 5% sample, sampling noise cancels — so any difference would be the restructure's fault. There were none: 66/66 figures pixel-identical, 26/26 numbers exact. The move is provably safe.

1 Why restructure — the problems we started with

The starting point (branch base bc60bad) had a working analysis but several structural hazards, the kind that Gentzkow & Shapiro's Code and Data for the Social Sciences warns against. The guiding goals were, in priority order: Safe (a move must never silently change a number), Clear (read the run order from the tree alone), Reproducible (one command rebuilds everything), Portable (no machine-specific paths), and Economic (the layout mirrors the paper's argument).

⚠ Hazards in the old layout

  • Machine-specific absolute paths — every script hard-coded F:/, U:/, or a C:/…/OneDrive/… root.
  • setwd() everywhere — each stage changed the working directory, so order & cwd mattered and breakage was silent.
  • Code & outputs mixed; figures collected through ad-hoc paths; no separation of "slow estimation" from "fast plotting".
  • No master script — no Makefile, .Rproj, or run_all; stages A–I were run by hand.
  • A fragile tmp/ hand-off between gabriel.R and pure.R (open issue O1).
  • Legacy mixed with canonical — abandoned files (F1–F7, G6, I5) sat next to the live ones.
  • CRLF/LF phantom diffs — 181 files could show as "modified" from line endings alone.
  • constants.R existed but was never wired in — magic numbers (γ, replacement rates, thresholds) were retyped across files.

✓ What the restructure delivers

  • One path file — config/paths.R: a single PATHS list; a folder move now edits one file, not forty.
  • Zero setwd() in canonical stages; the project root is found via a .here sentinel.
  • Functional split — build/ (make the data) → analysis/ (estimate) → presentation/ (the deck).
  • Three master scripts + a root RUN.R front door — source() one file to run a phase.
  • Persistent hand-off — the tmp/ seam routed to PATHS$analysis_temp.
  • Legacy quarantined in legacy/, each file guarded by a loud stop().
  • .gitattributes (* text=auto) kills the line-ending churn.
  • constants.R wired in — γ, η, replacement-rate schedules, thresholds defined once.

2 Before → After, at a glance

The reorganisation is a pure relocation + plumbing change: 314 of the file changes are git mv renames (history fully preserved), and not a single line of estimation logic was altered. Where the old code said setwd("F:/…"); fread('data/x.csv'), the new code says fread(file.path(PATHS$data_root, 'data', 'x.csv')) — same bytes read, but now portable.

3 The method — an orchestrator driving a team of agents

A reorganisation like this is risky precisely because it touches everything. The work was structured as five stages with a hard human-approval gate in the middle, and the labour was split between a single orchestrator (which held the plan and made decisions) and delegated sub-agents (which did the producing — mapping, running, repathing, verifying). The orchestrator never interleaved a folder move with a logic edit, so a parity failure would always have exactly one possible cause.

Why split into a team? One agent cannot line-level-map ~20 scripts (several over 1,000 lines) and run the 928 MB-panel pipeline well — too much context, and it mixes static reading with live execution. So mapping fanned out by code area, a conciliator stitched the pieces into one dependency graph, and a separate runner captured the baseline. The orchestrator only controlled; it never produced.

A conciliator merged the three partial maps into one canonical dependency graph and decided the parity set — exactly which stages actually run on the 5% sample.

Because the deck embeds a \today build-date on every Beamer frame, the holistic PDF hash always differs by date — so parity was anchored on the date-free per-figure PNG hashes and the numbers, not the raw PDF.

5 The portability layer — config/paths.R

This single file is the most important fix. It finds the project root with no setwd (via a .here sentinel), resolves the data environment loudly (an explicit error if no data root is found — never a silent wrong path), and exposes one PATHS list that every stage uses. A folder move now changes this file, not forty scripts.

# config/paths.R  (abridged)
PROJECT_ROOT <- rprojroot::find_root(rprojroot::has_file(".here"))   # no setwd, ever

# DATA_MODE: env override → detect by existing roots → loud stop()
if (Sys.getenv("PENSION_DATA_MODE") %in% c("full","sample")) DATA_MODE <- ...
else if (dir.exists(SAMPLE_ROOT)) DATA_MODE <- "sample"
else if (dir.exists(FULL_ANALYSIS_ROOT)) DATA_MODE <- "full"
else stop("could not resolve DATA_MODE — no recognised data root")   # never guess

PATHS <- list(
  data_root   = DATA_ROOT,          sample_data = file.path(SAMPLE_ROOT, "data"),
  output_F    = file.path(DATA_ROOT,"output","F"),   analysis_temp = file.path(DATA_ROOT,"tmp"),
  figures_from_code = file.path(PROJECT_ROOT,"presentation","figures_central_folder","from_code"),
  ... )                              # one list, every root + dir

run_stage(file)   # source a stage, timed + logged, variables don't leak
clear_dirs(...)   # wipe temp before a build (G–S "clear before you build")

A sibling config/constants.R holds every economic primitive once — γ (CRRA) = 4, the replacement-rate schedules (0.69 + 0.021·p women, 0.82 + 0.025·p men), thresholds (85 / 95), the bunching window, the DiD reference period, and the welfare weight η ≈ 0.828 computed from consumption levels rather than retyped. The LaTeX-to-code mapping is now one-to-one and auditable.

7 Stage 2 — repath & wire constants (commits 1ea2dd8, ae0c299)

Fourteen parallel agents — one per unit — performed plumbing only: drop every setwd, add the two source(config/…) lines, route all I/O through PATHS, substitute the safe constants (values are identical, so outputs cannot change), and flag — never fix the econometric bugs.

Path migration

setwd("F:/…") + fread('data/x') → fread(file.path(PATHS$data_root,'data','x')). The O1 hand-off rerouted to PATHS$analysis_temp, consistent on both the gabriel (write) and pure (read) sides.

Constant wiring

Literals like 85/95, γ=4, the RR coefficients → named constants. The discount literals 1.005 / 0.995 were left verbatim (that asymmetry is a flagged bug, not ours to "harmonise").

Static-checked

Every file parse()d / py_compiled — no heavy compute here. The real test was deferred to Stage 3's full re-run.

The ESC incident & recovery. Mid-run, the first Stage-2 agent batch was interrupted (an accidental Esc) after 13 of 14 units finished. Rather than redo good work, the 13 verified units were committed, and a focused recovery workflow finished the last unit — the four presentation Python tools. It even caught that the interrupted run had left update_deck.py half-edited (a dangling unused import os) and completed it.

9 The pipeline it all runs — the canonical DAG

Reading top-to-bottom, the new tree narrates the paper's economic argument: administrative records → eligible-worker sample & contribution histories → claiming behaviour & the counterfactual under pure reforms → DiD on average benefits → policy elasticity → MVPF/WMVPF welfare evaluation → the deck.

11 Code hygiene — used vs. archived

The functional move grouped files by area, but did not by itself separate "what the workflow runs" from superseded predecessors. A follow-up usage audit (two agents tracing every file's output consumers) fixed that — answering, for every file, is anything live actually consuming this? — so the folders are now legible without reading CLAUDE.md.

What changed

  • 11 confirmed-dead predecessors (E1–E3, G1–G3, H1, I1–I3, claiming2) → legacy/superseded/, each guarded with stop().
  • Every "dead" verdict was proven by grepping the file's output names and finding no live consumer.
  • A code/README.md manifest now labels every file in build/code/ and analysis/code/.
  • Nothing deleted — all git mv (reversible). Parity re-confirmed: no moved file is on any master.

⚠ The H2-vs-H3 puzzle (flagged)

CLAUDE.md calls H3 the canonical H stage — but H2 is the one that actually runs: H2 produces the elasticity table read by I4/I6 and the deck's elasticity figure, while H3 is full-data-only, figures-only, and on no master. Per Arthur, H2 is the real H stage; the CLAUDE.md label needs reconciling. H3 was kept in place (not archived) and the discrepancy recorded as h2-vs-h3 in the findings.

12 How to run it now

Two entry styles, same result — pick whichever you prefer:

  • Source a script (R / RStudio, or a batch job on the server): open R at the repo root and source() a master. RUN.R is the front-door signpost that lists them.
  • Terminal: run a master with Rscript from the repo root.

Either way there is no setwd and no path editing — config/paths.R finds the project root (via the .here sentinel) and resolves the data environment automatically. You only choose sample vs server through one env var (or it auto-detects by which data root exists).

🟢 With the sample (local, 5% data)

Validation on your machine. The 928 MB panel + cross-section are pre-supplied in the external sample root, so the build stage is skipped (the panel is an input, not rebuilt).

From R / RStudio — the script way:

# open R at the repo root, then:
source("RUN.R")                                   # signpost: prints the 3 entry points
source(here::here("analysis","analysis_all.R"))   # panel → figures, tables, WMVPF
source(here::here("presentation","build_deck.R")) # figures → English deck (_main.pdf)

From the terminal:

Rscript analysis/analysis_all.R
Rscript presentation/build_deck.R

Auto-detected as DATA_MODE=sample when the sample root exists; point it elsewhere with PENSION_SAMPLE_ROOT=…/transfer_may_retirement. (build/build_all.R intentionally stop()s here — the panel is an input on the sample.)

🔵 On the server (full SUIBE/RAIS)

The real pipeline on the restricted-access server — here you also rebuild the panel from raw data before estimating. Set the mode once, then run all three masters in order.

From R / RStudio — the script way:

# in R, at the repo root:
Sys.setenv(PENSION_DATA_MODE = "full")            # force full-data mode
source(here::here("build","build_all.R"))         # raw SUIBE/RAIS → analysis-ready panel
source(here::here("analysis","analysis_all.R"))   # panel → estimates, figures, WMVPF
source(here::here("presentation","build_deck.R")) # figures → deck

Or as a shell / batch job:

export PENSION_DATA_MODE=full
Rscript build/build_all.R         # full-data only (stops unless DATA_MODE=full)
Rscript analysis/analysis_all.R
Rscript presentation/build_deck.R

From-scratch rebuild: build_all.R runs the canonical heads A4/B4/C6/D4; the upstream lineage (A1–A3, B1–B3, C1–C5, D1–D3) that fills working/ must run first — see build/code/README.md. Stata steps (B1–B3, C3) run with stata-mp -b do build/code/<file>.do.

The analysis master also runs a prerequisite check and stops if the tables it consumes (legacy F5, siblings G4/H2) are missing, rather than producing a wrong number.
