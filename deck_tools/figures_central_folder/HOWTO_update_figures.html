From code to slide: updating the presentation's figures

A visual guide for someone who has R, Python and LaTeX installed, but is new to this repository.

✅ Short answer: yes, it is automatic

If you change a script that makes a figure, you update that figure in the slides with one command:

python deck_tools/figures_central_folder/update_deck.py G5
# reruns the G5 script → collects its figures → rebuilds the slides PDF

Replace G5 with the stage you changed (E4, G5, H2, F, Fg, I6). The rest of this page explains what that command does and how the pieces connect — so you can trust it and fix it if something looks off.

analysis code figure files the code writes central hub (the connector) the presentation

1 The four places that matter

You only need to know four locations. Everything in this guide lives in one of them.

# the repository root: optimal-pension-reforms/
├─ trans_retirement/code/ ← the R scripts that DO the analysis & draw the plots
│     └─ G5_effect_average_benefit_freq_bL_and_bS.R (makes the benefit plots)
├─ trans_retirement/output/ ← figures land here when run on the SERVER (full data)
├─ figures_central_folder/ ← THE CONNECTOR (this folder)
│     ├─ manifest.csv — the switchboard: which plot feeds which slide
│     ├─ collector.py — copies plots in, renames them for the slides
│     ├─ update_deck.py — the one-command workflow
│     └─ from_code/ — the plots, ready for the slides (auto-filled)
└─ latex/presentation/ ← the Beamer slides
       └─ _main.tex → _main.pdf

# + one place OUTSIDE the repo:
…/OneDrive/…/transfer_may_retirement/ ← the 5% SAMPLE data + where figures land when run LOCALLY
💡
Why two output places? The analysis can run on the real full data (only on the secure server → writes to trans_retirement/output/) or on a small 5% sample on your laptop (→ writes into the OneDrive transfer_may_retirement folder). The connector knows how to read from either.

2 How one figure travels from code to a slide

Follow a real example — the slide “Effect on Avg Actual Benefits at p∈[0,2)”. Its picture is called new0.pdf in the slides.

① you edit / run the script
trans_retirement/code/G5_effect_average_benefit_freq_bL_and_bS.R
This R script runs the regression and draws the plot.
▼run it (R)
② the script writes a PDF
…/transfer_may_retirement/output/G/G4_eventstudy_benegits_new_3.pdf (sample run · on the server it would be trans_retirement/output/G/…)
Note the code's name is long and technical — not what the slide asks for.
▼collector.py — copy & rename
③ the collector renames it for the slides
latex/figures/from_code/new0.pdf
The collector looks up the rename in manifest.csv (see §3) and copies the newest version of the plot here under the exact name the slides use.
▼\graphicspath points here
④ the slide shows it
latex/presentation/_main.tex  →  \includegraphics{new0.pdf}
Recompile _main.tex and the slide now shows the fresh plot.
🔑
The slide file never changed — it always asked for new0.pdf. We just made sure the right, current new0.pdf shows up in from_code/. That is the whole trick.

3 The switchboard: manifest.csv

Every figure in the slides has exactly one row in manifest.csv. That row says which script makes it, where the file is, and what the slide calls it. This is the link between code and presentation.

columnfor our example (new0.pdf)meaning
deck_namenew0.pdfthe name the slide uses
code_scriptG5_…bL_and_bS.Rwhich script produces it
code_output_pathtrans_retirement/output/Gthe folder the code writes to
code_output_nameG4_eventstudy_benegits_new_3.pdfthe code's own filename
statusOK-RENAMEOK / OK-RENAME / LEGACY / NONE …
modeanysample-only? full-only? either?

So to find out where any slide's picture comes from, open manifest.csv and search for its name. To re-point a slide to a different plot, change that one row — no LaTeX editing needed.

4 Update a figure — the one-command way

You edited G5 and want the slides to reflect it. Run:

python deck_tools/figures_central_folder/update_deck.py G5

That command does the three steps of §2 for you:

stepwhat it runsresult
1 · runRscript …/G5_…Rregenerates the plot file
2 · collectcollector.py --sample-root …copies+renames it into from_code/
3 · compilelatexmk -pdf _main.texrebuilds _main.pdf

Known stage shortcuts: E4 (claiming plots), G5 (benefit plots), H2 (tax plots), F & Fg (frequency plots), I6 (WMVPF plot). You can also pass several at once: update_deck.py G5 H2.

💡
Just changed the slides or a manifest row, but didn't touch any R code? Skip the slow R step: 
python deck_tools/figures_central_folder/update_deck.py --collect-only

5 The same thing, step by step (no wrapper)

If you prefer to see each piece, run them yourself from the repo root:

# 1 · regenerate the figure (writes into the sample folder)
Rscript trans_retirement/code/G5_effect_average_benefit_freq_bL_and_bS.R

# 2 · pull the fresh figures into the central folder
python deck_tools/figures_central_folder/collector.py \
       --sample-root "C:/Users/<you>/OneDrive/Documentos/Pesquisa/transfer_may_retirement"

# 3 · rebuild the slides
cd latex/presentation && latexmk -pdf -interaction=nonstopmode _main.tex

Check the figure landed: python deck_tools/figures_central_folder/verify_deck.py should say unresolved: 0.

6 Add a NEW figure to the presentation

Adding a brand-new picture is three small moves: tell the switchboard where it comes from (one row in manifest.csv), put it on a slide (\includegraphics), then run the workflow. First, which kind of figure is it?

made by a script (a plot) made by hand (diagram, screenshot, one-off)

① Choose the name the slide will use

Pick a short deck name, e.g. my_new_plot.pdf. That is the name you write in the slide and the name the file gets inside the central folder. It does not have to match the code's own filename — the switchboard handles the rename.

② Add one row to manifest.csv

The columns are always deck_name, code_script, code_output_path, code_output_name, status, mode, notes.

A · the figure is produced by a script — say you added a new ggsave('output/G/G5_my_panel.pdf', ...) to the G5 script:

# deck_name, code_script, code_output_path, code_output_name, status, mode, notes
my_new_plot.pdf,G5_effect_average_benefit_freq_bL_and_bS.R,trans_retirement/output/G,G5_my_panel.pdf,OK-RENAME,any,My new benefit panel

B · the figure is hand-made / external — a diagram or screenshot with no code behind it. Drop the file into latex/figures/static/, then add:

# deck_name, code_script, code_output_path, code_output_name, status, mode, notes
reform_sketch.pdf,(manual/external),latex/figures/static,reform_sketch.pdf,NONE,n/a,Hand-made sketch
statususe it when…
OK-RENAMEa script makes it under a different filename (the usual case for plots)
OKa script makes it and you keep the code's exact filename as the deck name
NONEno code makes it — the file lives in static/

mode: any for a normal plot, sample if it only exists from a 5% run, full if only from the server, n/a for a hand-made file.

③ Put it on a slide in latex/presentation/_main.tex

\begin{frame}{My new slide}
  \centering
  \includegraphics[width=0.8\linewidth]{my_new_plot.pdf}
\end{frame}

Write the deck name only — no folder path. The \graphicspath already points at the central folder, so LaTeX finds it.

④ Run the workflow

# A · a script makes it — rerun that stage so the file exists, then collect + compile:
python deck_tools/figures_central_folder/update_deck.py G5

# B · hand-made (already in static/), or the file already exists — just collect + compile:
python deck_tools/figures_central_folder/update_deck.py --collect-only

Using a script that isn't one of the shortcuts? Pass its path: update_deck.py trans_retirement/code/MyScript.R. Then confirm it landed: verify_deck.py → unresolved: 0.

⚠️
No manifest row = the figure is invisible. The collector only moves figures that are listed in manifest.csv. If a new slide shows blank or “file not found”, the usual cause is a missing (or mistyped) row. That one row is the entire connection between your figure and the slides.
✅
Worked example. Add ggsave('output/G/G5_my_panel.pdf') to G5 → add the row from A → put \includegraphics{my_new_plot.pdf} on a slide → run update_deck.py G5. The new slide now shows your plot, straight from the code.

7 The sample-vs-server detail (this was the “broken” part)

Each analysis script auto-detects where it is running and writes its figures accordingly:

running…datafigures land in…collector reads them via…
on your laptop5% samplethe OneDrive transfer_may_retirement/output/--sample-root <that folder>
on the secure serverfull datatrans_retirement/output/nothing extra (the default)

The connector reads from both places and keeps the newest copy. That is why a stale figure in the repo no longer overrides a fresh one you just produced on the sample. This was the bug: before, the slides were pulling old/hand-copied pictures that had drifted away from the code.

⚠️
5% sample plots are noisy. They are perfect for checking the pipeline works, but they are not publication quality. The polished versions come from a full-data run on the server — same command, just no --sample-root. The collector tags sample figures with [SAMPLE] so you never ship one by accident.

8 When a figure won't update — checklist

symptomwhat to check
slide still shows the old plotDid step 3 (compile) run? Try --collect-only then recompile. PDFs cache — make sure you reopened the new _main.pdf.
collector says MISSINGThe script didn't actually write that file. Re-run the stage; confirm the file exists at code_output_path (or in the sample folder).
collector picked an old copyPass --sample-root so it sees your fresh sample run. It keeps the newest of sample-vs-repo.
plot looks wrong / degenerateThat's the code's output, not a routing problem. Open the file named in manifest.csv and check the script. (A tiny PDF often means a near-empty sample cell.)
[LEGACY] warningThat figure comes from an old script with no current generator — treat it as possibly outdated.
it's a hand-made picturestatus = NONE figures (diagrams, screenshots) live in latex/figures/static/ and are not produced by code.

9 Cheat sheet

# one-command update after editing a stage
python deck_tools/figures_central_folder/update_deck.py G5

# just re-route + recompile (no R)
python deck_tools/figures_central_folder/update_deck.py --collect-only

# check every slide figure resolves
python deck_tools/figures_central_folder/verify_deck.py

# see what changed between the old and new slides (page-by-page)
python deck_tools/figures_central_folder/deck_compare.py OLD.pdf NEW.pdf figures_central_folder/_diffs

# find where a slide's picture comes from
→ open figures_central_folder/manifest.csv and search the figure name
